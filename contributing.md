## 🛠️ 技術貢獻流程

### 🎯 **Fork 和克隆**

#### 1. Fork 專案
```bash
# 在 GitHub 上點擊 Fork 按鈕
# 或者使用 GitHub CLI
gh repo fork yourusername/opencode-guide
```

#### 2. 克隆你的 Fork
```bash
git clone https://github.com/yourusername/opencode-guide.git
cd opencode-guide
```

#### 3. 添加上游倉庫
```bash
git remote add upstream https://github.com/original/opencode-guide.git
git fetch upstream
```

### 🌿 **分支管理**

#### **創建功能分支**
```bash
# 確保在最新的 main 分支
git checkout main
git pull upstream main

# 創建新的功能分支
git checkout -b feature/improve-installation-guide
```

#### **分支命名規範**
```bash
# 功能開發
feature/feature-name

# 錯誤修復
fix/issue-description

# 文檔更新
docs/update-section-name

# 重構
refactor/component-name
```

### 📝 **內容編輯**

#### **文檔格式標準**
```markdown
# 標題使用 H1 (只有一個)
## 節節使用 H2
### 子節節使用 H3
#### 詳細內容使用 H4

- 使用項目符號列表
1. 使用有序列表

`inline code` 使用反引號

```code block
```

> 引用內容

![圖片描述](path/to/image.png)

[連結文字](URL)
```

#### **代碼範例標準**
```typescript
// 添加語言標識
// 包含完整的代碼範例
// 添加適當的註釋

## 📋 內容貢獻指南

### ✅ **優質內容標準**

#### **準確性要求**
- [ ] **技術準確** - 所有技術內容必須經過驗證
- [ ] **版本相容** - 標明相容的 OpenCode 版本
- [ ] **平台相容** - 標明支援的作業系統
- [ ] **範例可執行** - 所有代碼範例必須可以執行

#### **完整性要求**
- [ ] **上下文完整** - 提供充分的背景資訊
- [ ] **步驟清晰** - 每個步驟都詳細說明
- [ ] **預期結果** - 明確說明預期結果
- [ ] **錯誤處理** - 包含常見錯誤和解決方案

#### **可讀性要求**
- [ ] **語言簡潔** - 避免冗長和複雜的表達
- [ ] **結構清晰** - 邏輯層次清楚
- [ ] **術語統一** - 使用統一的術語和概念
- [ ] **格式一致** - 遵循既定的格式標準

### 🎯 **內容優先級**

#### **高優先級內容**
1. **基礎功能說明** - 核心功能的詳細說明
2. **入門指南** - 新手的完整入門流程
3. **常見問題** - 用戶最常遇到的問題
4. **實戰案例** - 實際使用場景的完整範例

#### **中優先級內容**
1. **進階功能** - 高級用戶的專業功能
2. **最佳實踐** - 推薦的使用方式和技巧
3. **故障排除** - 詳細的問題解決指南
4. **性能優化** - 效能提升的具體方法

#### **低優先級內容**
1. **邊緣案例** - 特殊情況的處理方式
2. **歷史背景** - 功能的發展歷史
3. **比較分析** - 與其他工具的對比
4. **未來規劃** - 功能的發展方向

## 🧪 質量保證

### ✅ **檢查清單**

#### **內容檢查**
```markdown
# 提交前檢查清單
- [ ] 內容準確無誤
- [ ] 代碼範例可以執行
- [ ] 格式符合標準
- [ ] 連結有效可訪問
- [ ] 圖片顯示正常
- [ ] 拼寫和語法正確
- [ ] 版本資訊正確
- [ ] 標籤和分類合適
```

#### **技術驗證**
```bash
# 測試代碼範例
npm test
node examples/hello-world.js

# 檢查文檔構建
npm run docs:build
npm run docs:serve

# 測試連結有效性
npm run check-links
```

#### **預覽測試**
```bash
# 本地預覽
npm run docs:dev
# 訪問 http://localhost:3000

# 移動端測試
npm run docs:build
npm run docs:preview-mobile
```

### 🔄 **審查流程**

#### **自我審查**
1. **內容審查** - 檢查內容的準確性和完整性
2. **格式審查** - 確保格式符合專案標準
3 **功能測試** - 驗證所有範例和連結
4. **用戶體驗** - 從用戶角度檢查可讀性

#### **同行審查**
1. **PR 描述** - 提供清晰的 PR 描述
2. **變更說明** - 詳細說明每次變更的原因
3. **測試證據** - 提供測試通過的證據
4. **回應反饋** - 及時回應審查者的意見

## 🚀 Pull Request 流程

### 📝 **PR 創建標準**

#### **PR 標題格式**
```bash
# 功能添加
feat: 添加新的配置選項說明

# 錯誤修復
fix: 修復安裝指南中的命令錯誤

# 文檔更新
docs: 更新最佳實踐章節

# 重構
refactor: 重構代碼範例結構

# 樣式調整
style: 統一代碼塊格式
```

#### **PR 描述模板**
```markdown
## 🎯 變更類型
- [ ] Bug 修復
- [ ] 新功能
- [ ] 文檔改進
- [ ] 代碼重構
- [ ] 性能優化
- [ ] 其他

## 📝 變更描述
簡要描述這次變更的內容和目的

## 🔧 實現方式
說明如何實現這次變更

## 🧪 測試
描述如何測試這次變更
- [ ] 本地測試通過
- [ ] 文檔構建成功
- [ ] 連結檢查通過

## 📷 截圖（如適用）
如果涉及 UI 變更，請提供截圖

## 🔗 相關 Issue
關聯的 Issue 編號：#123

## ✅ 檢查清單
- [ ] 我的變更遵循專案的代碼規範
- [ ] 我已經自我審查了這次變更
- [ ] 我已經添加了必要的文檔
- [ ] 新的變更已經添加了測試
- [ ] 所有測試都通過了
```

### 🔄 **審查和合併**

#### **審查者注意事項**
1. **技術準確性** - 驗證技術內容的正確性
2. **用戶價值** - 評估變更對用戶的價值
3. **一致性** - 檢查與現有內容的一致性
4. **可維護性** - 評估內容的長期維護性

#### **合併標準**
- [ ] 至少一個審查者通過
- [ ] 所有檢查項目都通過
- [ ] 沒有衝突或已解決衝突
- [ ] 文檔構建成功
- [ ] 關聯 Issue 已關閉（如有）
